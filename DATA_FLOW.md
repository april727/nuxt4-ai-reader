# AI Reader 数据流与 DeepSeek 调用说明

## 完整数据流

```
导入 → 分析+分段 → 阅读标记 → 复习本 → 单词本 → 闪卡学习
  │       │           │          │         │         │
  │   2次API调用   每词1次     0次      0次       0次
  │               API调用    API调用  API调用   API调用
  ▼
 文本存储
（零API）
```

---

## 一、导入（零 API 调用）

**触发**：用户拖拽文件、粘贴文本、输入网址

**流程**：
- PDF → PyMuPDF / pdfjs-dist 提取文字 → 存入 texts 表
- 网页 → Mozilla Readability 提取正文 → 去重 → 存入 texts 表
- 粘贴 → 直接存入 texts 表

**不调 DeepSeek**。纯文本提取 + SQLite 存储。

---

## 二、AI 分析 + 分段（2 次 API 调用）

**触发**：用户点击阅读页顶部「AI 分析」或「重新分段」按钮（不再自动触发）

**第一步 — 流式分析**（1 次 `streamExplain` 调用）：
- 截取全文前 **15000 字符**发给 DeepSeek
- 要求生成：标题、风格、内容摘要、背景分析、关键要点
- 结果实时流式输出到右侧面板

**第二步 — 分段**（1 次 `segment` 端点调用）：
- 同样截取前 **15000 字符**
- 智能识别内容类型（书籍/对话/教程/议论文），多信号定位段落边界
- 视频字幕额外做去重和时序映射
- 失败时本地兜底：每 4 句合并为一段

**Token 消耗**：15000 字符 ≈ 3000-5000 tokens，远低于 128K 上下文上限。

---

## 三、阅读标记（每词 1 次 API 调用）

**触发**：用户选中文字 → 点击「词」「短」「句」按钮

**流程**：
```
选中文字 → handleCreateMark()
  → streamExplain(流式) → AI 返回结构化解释 → 存入 mark.detail
  → saveMarks() → text/update → 自动同步到单词本
```

**发给 AI 的内容**（不是全文，非常精简）：

| 内容 | 来源 | 大小 |
|------|------|------|
| 文章背景 | AI 分析生成的摘要（标题+摘要+背景+要点） | ~300 字 |
| 所在段落 | 当前段落的全部文本 | ~200-500 字 |
| 待解释词 | 选中的文本 | 1-50 字 |
| 提示词 | 类型相关的格式要求 | ~200 字 |

**Token 消耗**：每次标记约 400-800 tokens。全文不发给 AI，文章背景用摘要替代。

**AI 返回格式**（以生词为例）：
```markdown
### 基本释义
跑步；运行
### 其他常用用法
...
### 当前语境用法
...
[PHONETIC]/rʌn/[/PHONETIC]
[LEMMA]/run/[/LEMMA]
```

**关键时序**：标记时先创建本地 mark 对象（`detail: ''`），等 AI 流式返回完成后填入 detail，再调用 `saveMarks()` 持久化。保证同步到单词本的数据是完整的。

---

## 四、复习本（零 API 调用）

**触发**：打开复习本页面 `/reviews`

**流程**：
- `GET /api/reviews/list` → 扫描所有 texts 表的 marks JSON 字段
- 扁平化每个 mark 为 `{ mark, title, textId, phonetic, brief }` 结构
- 纯 SQL 查询 + JSON 解析，不调 AI

**页面功能**：
- 按类型筛选（生词/短语/好句）
- 按来源书籍筛选
- 快捷入口直达三个默认单词本
- 导出 Anki CSV / Markdown

---

## 五、单词本同步（零 API 调用）

**触发**：自动，标记保存时（`text/update` 端点的副作用）

**流程**：
```
mark 保存 → text/update.post.ts
  → syncMarksToWordbooks():
    1. diff 新旧 marks → 找出新增的
    2. word → wb_default（默认单词本）
       phrase → wb_phrases（默认短语本）
       sentence → wb_sentences（默认句子本）
    3. 从 mark.detail 中正则提取音标、含义、原词
    4. 去重后写入 words 表
```

**数据提取规则**：

| 字段 | 提取来源 | 正则 |
|------|---------|------|
| phonetic | `[PHONETIC]/.../[/PHONETIC]` | 标签匹配 |
| meaning | `### 基本释义` 段落 | 取到下一标题或标签前 |
| lemma | `[LEMMA]/.../[/LEMMA]` 或 mark.lemma | 标签匹配 |

**原词优先**：如果 AI 返回了原词（如 running→run），单词本以原词为主词条，原文变形存入 note。

**不调 DeepSeek**。数据全部来自标记时 AI 已生成并存储的 `mark.detail`。

---

## 六、单词本补全（极少 API 调用）

**触发**：用户点击词条行尾 🤖 或顶部「一键补全」

### 批量补全（一键全部）

```
enrich-batch.post.ts:
  1. 扫描所有 texts 的 marks → 建立 {词文本 → {音标,含义,原词}} 索引
  2. 遍历单词本无含义词条 → 查索引 → 直接写入
  3. 零 DeepSeek 调用
```

### 单个补全（行尾按钮）

```
enrich.post.ts:
  1. 先查已有 marks 索引 → 找到 → 直接返回（零 API）
  2. 找不到（手动添加的词）→ 调 DeepSeek 兜底（1 次 API）
```

### TXT 导入

```
import.post.ts:
  每 5 个词一组调 DeepSeek → 返回 JSON → 批量写入
  { "words": [{"word":"run","phonetic":"/rʌn/","meaning":"跑步","example":"..."}] }
```

---

## 七、闪卡学习（零 API 调用）

**触发**：进入 `/wordbooks/[id]/cards`

**流程**：
- 加载词条列表 → 过滤已掌握的 → 构建学习队列
- 正面显示单词 → 点击/空格翻转 → 显示释义 → 评级（忘记/困难/良好/简单）
- SM-2 间隔重复算法计算下次复习时间
- 全部在客户端运行，不调 AI

---

## DeepSeek 调用次数汇总

以一篇 10 段文章、标记 20 个生词为例：

| 操作 | 次数 | 类型 | 说明 |
|------|------|------|------|
| 导入 | 0 | - | 纯文本提取 |
| AI 分析 | 1 | 流式 | 截断 15000 字符 |
| AI 分段 | 1 | 普通 | 截断 15000 字符 |
| 标记 20 词 | 20 | 流式 | 每次约 400-800 tokens |
| 查看复习本 | 0 | - | 纯 SQL 查询 |
| 同步到单词本 | 0 | - | 提取已有数据 |
| 一键补全 | 0 | - | 查 marks 索引 |
| 闪卡学习 | 0 | - | 纯本地 |
| **合计** | **22** | | |

---

## Token 消耗控制策略

| 场景 | 策略 | 每次约消耗 |
|------|------|-----------|
| AI 分析 | 截取前 15000 字符 | 3000-5000 tokens |
| AI 分段 | 截取前 15000 字符 | 3000-5000 tokens |
| 标记 | 只发摘要+段落+词，不发全文 | 400-800 tokens |
| 段落解释 | 只发段落+文章摘要 | 500-1000 tokens |
| 单词本补全 | **不发 AI**，从已有 marks 提取 | 0 tokens |
| 单词本导入 | 每 5 词一组 | 200-400 tokens/组 |

---

## 三个默认单词本

| ID | 名称 | 对应标记类型 | 创建方式 |
|----|------|-------------|---------|
| `wb_default` | 默认单词本 | word（生词） | 服务启动自动创建 |
| `wb_phrases` | 默认短语本 | phrase（短语） | 服务启动自动创建 |
| `wb_sentences` | 默认句子本 | sentence（好句） | 服务启动自动创建 |

默认本不可删除，排序置顶，在复习本页面有快捷入口。
